# 项目完善与迭代路线图
> 基于当前仓库真实实现整理，不按理想化平台设计展开，优先保证现有主链路稳定，再逐步补强 Agent 能力和平台能力。

## 1. 当前项目定位

当前项目的核心定位已经比较明确：

1. 用户输入自然语言需求，创建一个应用
2. Agent 根据需求生成或修改应用代码
3. 前端在对话页中展示生成结果和预览页面
4. 用户可以继续对话迭代、下载代码、部署应用
5. 平台提供基础的作品展示、点赞、分享、评论和后台管理能力

也就是说，当前项目更像一个：

**面向网页/小应用生成场景的 Agent 应用生成平台**

而不是一个完全通用的 Agent 开发框架。

所以后续完善时，应该优先服务这个主目标，避免一开始就把精力分散到过度抽象的平台化改造上。

---

## 2. 当前架构认知

## 2.1 前端

前端位于 `app-creator-fronted/app-creator-fronted`

当前主要页面职责：

- `HomePage.vue`
  - 输入需求并创建应用
  - 展示“我的作品”和“精选应用”
- `AppChatPage.vue`
  - 核心工作台
  - 左侧对话，右侧预览
  - 支持继续修改、下载、部署、点赞、分享
  - 已经具备“可视化选中页面元素再发需求”的雏形
- 管理页
  - 用户管理
  - 应用管理
  - 对话管理

前端特点：

- 已经形成了完整业务闭环
- 通过 OpenAPI 自动生成接口代码，前后端联动成本较低
- 聊天页承载了项目最核心的用户体验

## 2.2 后端

后端位于 `src/main/java/com/xiaorui/agentapplicationcreator`

当前主要模块：

- `controller`
  - 用户、应用、Agent、任务、评论、点赞、分享等接口
- `service`
  - 业务实现层
- `agent`
  - Agent 生成、编排、任务执行、plan 校验执行、工具、RAG、sub-agent 等
- `mapper`
  - MyBatis-Flex 数据访问
- `security / manager / util / config`
  - 鉴权、限流、缓存、对象存储、部署、截图、通用配置

后端特点：

- 业务链路已经串起来了
- 不只是 CRUD，已经包含异步任务、线程绑定、部署、截图、限流、评论互动等能力
- Agent 相关能力很多，但成熟度存在明显差异，有些是主路径，有些是预留能力

## 2.3 主链路

当前最关键的主链路是：

1. 首页创建应用
2. 进入应用聊天页
3. 发送消息给 Agent
4. 创建异步任务并轮询结果
5. 落盘生成代码
6. 前端加载预览
7. 用户继续修改
8. 下载或部署

后续所有迭代，默认都应该围绕这条链路做增强，而不是脱离这条链路单独扩展。

## 3. 当前项目的主要问题判断

这里不从“代码规范是否完美”角度看，而从“下一步怎么做最有效”角度看。

## 3.1 已经具备的优势

- 主业务闭环已经跑通
- 前后端分离明确，接口边界比较清楚
- Agent 已经不是简单聊天，而是和应用、文件、任务、历史、部署绑定在一起
- 有任务状态、失败重试、thread 绑定等平台雏形
- 有较多扩展点，后续可持续演进

## 3.2 当前主要短板

- 核心链路的稳定性和一致性还需要加强
- 配置项较分散，环境依赖较重
- Agent 返回结构、任务状态、文件落盘、预览刷新之间还存在耦合和脆弱点
- 安全边界不够收紧，尤其是鉴权、跨域、文件操作、部署链路
- 文档设计比当前落地更超前，部分模块是“有设计但未完全收口”
- 前端聊天页承担的职责很多，后续继续加功能会变得更难维护

## 3.3 需要避免的方向

- 不要立刻做大规模重构
- 不要先做太多“通用平台抽象”
- 不要先做大量边缘功能
- 不要在主链路还不够稳时继续叠加复杂 Agent 能力

---

## 4. 迭代总原则

后续完善遵循以下原则：

1. 先稳主链路，再扩能力
2. 先收边界，再谈抽象
3. 先提升可维护性，再补高级特性
4. 所有新功能尽量复用当前已有模块，不轻易推翻重写
5. 每个阶段都应保证“项目仍然能跑、主功能仍然可用”

---

## 5. 分阶段路线图

建议按 `P0 -> P1 -> P2 -> P3` 推进。

## 5.1 P0：主链路加固

这一阶段最重要，优先级最高。

目标：

**保证“创建应用 -> 对话生成 -> 预览 -> 修改 -> 下载/部署”这条链路稳定、可追踪、可恢复。**

### 需要完成的事项

#### 1. 统一任务状态与返回约定

重点模块：

- `AgentController`
- `AgentTaskController`
- `DefaultAgentOrchestrator`
- `AgentTaskServiceImpl`
- 前端 `AppChatPage.vue`

目标：

- 明确任务状态流转
  - `WAITING`
  - `RUNNING`
  - `RETRY_WAITING`
  - `SUCCEEDED`
  - `FAILED`
- 明确轮询接口在各状态下返回什么字段
- 明确失败时前端展示什么信息
- 避免前端依赖过多隐式字段拼装状态

#### 2. 收紧代码生成与文件落盘链路

重点模块：

- `AgentAppCreator`
- `CodeFileSaverUtil`
- `agent.plan.*`
- `FileOperationTool`

目标：

- 明确当前主路径到底是“结构化文件列表落盘”还是“修改计划执行”
- 对文件输出目录、文件覆盖策略、异常处理做统一约束
- 防止路径异常、空返回、重复写入、半成功半失败状态

#### 3. 统一预览刷新机制

重点模块：

- 前端 `AppChatPage.vue`
- 前端 `env.ts`
- 后端静态资源预览相关控制器和目录结构

目标：

- 明确不同 `codeGenType` 的预览路径规则
- 避免前端预览逻辑和后端产物目录不一致
- 明确单文件模式、Vue 项目模式的预览入口
- 让“生成成功后自动刷新预览”更稳定

#### 4. 强化部署前校验

重点模块：

- `AppServiceImpl`
- `ProjectDownloadServiceImpl`
- `SftpFileUtil`

目标：

- 部署前检查生成产物是否存在且完整
- 失败时给出明确错误原因
- 区分“代码未生成”“拷贝失败”“SFTP 上传失败”“部署地址回写失败”

#### 5. 补足主链路日志和排错信息

重点模块：

- `DefaultAgentOrchestrator`
- `AgentAppCreator`
- `AppServiceImpl`

目标：

- 每次任务至少能追踪：
  - 用户
  - 应用
  - threadId
  - taskId
  - 状态变化
  - 失败原因
- 后续排查问题时，不需要依赖猜测

### P0 完成标准

- 新建应用后可稳定进入聊天页
- 首次生成成功后可看到可用预览
- 修改请求后能稳定更新预览
- 下载和部署失败时能快速定位原因
- 前端不会因为少数字段为空而直接失控

---

## 5.2 P1：配置治理与安全边界收口

目标：

**让项目从“本地可跑”逐步变成“环境可控、边界清晰、风险可管理”。**

### 需要完成的事项

#### 1. 整理环境配置

重点模块：

- `application.yml`
- `application-dev.yml`（后续补充）
- 各类 `Config`
- 前端 `env.ts` / `env.example.ts`

目标：

- 将模型、Redis、MongoDB、MinIO、SFTP、部署域名、静态预览目录等配置系统化
- 区分开发环境和生产环境
- 避免把路径、域名、端口写死在业务逻辑里

#### 2. 收紧鉴权和接口访问边界

重点模块：

- `AuthFilter`
- `WebSecurityConfig`
- `AuthConfigAdapter`
- `AuthCheck`

目标：

- 明确哪些接口必须登录
- 明确哪些接口允许匿名
- 减少“靠前端约束，后端默认放行”的情况
- 补充关键资源的权限校验

#### 3. 收紧跨域和预览资源策略

重点模块：

- `CorsConfig`
- 预览静态资源暴露方式

目标：

- 不再使用过宽的跨域策略作为长期方案
- 区分 API、预览资源、部署域名三种访问场景
- 为后续 iframe 预览和部署隔离做准备

#### 4. 收紧文件系统与部署边界

重点模块：

- `FileOperationTool`
- `CodeFileSaverUtil`
- 部署目录相关常量

目标：

- 统一项目产物目录结构
- 限制 Agent 可触达的目录边界
- 防止后续继续扩展后出现更高风险的文件写入问题

### P1 完成标准

- 环境变量和配置说明更完整
- 接口权限边界清晰
- 开发和生产环境切换成本降低
- 文件与部署路径不再散落在多处业务代码中

---

## 5.3 P2：Agent 能力完善

目标：

**在主链路稳定后，再提升 Agent 的可控性、上下文质量和可扩展性。**

### 需要完成的事项

#### 1. 梳理 Agent 输入输出协议

重点模块：

- `AgentAppCreator`
- `AgentResponse`
- `StructuredReply`
- prompt 资源文件

目标：

- 明确 Agent 必须返回哪些字段
- 区分“面向用户展示的回复”和“面向系统执行的结构化信息”
- 减少前端和后端对模型输出格式的脆弱依赖

#### 2. 完善上下文记忆策略

重点模块：

- `UserThreadBindServiceImpl`
- `ChatHistoryServiceImpl`
- Redis 记忆相关模块

目标：

- 明确谁是主记忆来源：MySQL、Redis 各自负责什么
- 避免同一份历史在多个存储中职责混乱
- 提高多轮修改时的上下文稳定性

#### 3. 完善计划执行能力

重点模块：

- `agent.plan.*`
- `PlanValidator`
- `PlanExecutor`

目标：

- 提升“根据页面元素修改现有代码”的可控性
- 让修改计划更可验证、更容易失败回滚
- 为后续多文件、多页面修改做准备

#### 4. 整理 sub-agent 和 code optimization 能力

重点模块：

- `agent.subagent.*`
- `CodeOptimizeResultServiceImpl`

目标：

- 明确它目前是在线主流程能力，还是离线增强能力
- 如果暂时不成熟，先降级为可选异步增强
- 避免主链路被不稳定优化链路拖垮

#### 5. RAG 接入前先做接口留白收口

目标：

- 先把知识检索的调用边界和数据结构定下来
- 不急着大规模做文档入库
- 保证后续接入时不需要重写当前主流程

### P2 完成标准

- Agent 输出结构更稳定
- 多轮修改体验更自然
- 页面元素选中修改能力更可靠
- sub-agent 和优化链路不会影响主流程稳定性

---

## 5.4 P3：平台能力扩展

目标：

**在核心产品成立之后，进一步补足“作品平台”和“持续创作平台”的体验。**

### 可选方向

#### 1. 应用版本管理

价值：

- 用户每次生成或修改后可以形成版本
- 支持版本回退、差异查看、快照恢复

#### 2. 应用模板化

价值：

- 把已有成功作品沉淀为模板
- 降低后续生成的不确定性

#### 3. 更完整的社区能力

价值：

- 评论通知
- 点赞/分享消息提醒
- 精选推荐机制
- 应用排行或分类

#### 4. 更完整的后台运营能力

价值：

- 任务监控
- 失败任务查看
- 应用质量巡检
- 用户行为统计

#### 5. 可观测性面板

价值：

- 任务成功率
- 平均生成时长
- 部署成功率
- 模型调用消耗

### P3 原则

- P3 必须建立在 P0 和 P1 足够稳的基础上
- 平台功能应优先复用已有表结构和业务链路
- 不为了“看起来功能多”而过早复杂化

---

## 6. 优先级建议

如果按当前项目状态排序，建议优先级如下：

### 第一优先级

- 主链路加固
- 任务状态统一
- 代码落盘和预览链路稳定
- 部署前校验

### 第二优先级

- 配置治理
- 权限和跨域边界收口
- 文件系统边界收口

### 第三优先级

- Agent 上下文与结构化输出优化
- 页面元素级修改能力增强
- sub-agent 能力整理

### 第四优先级

- 模板、版本、社区、监控、运营等平台功能

---

## 7. 推荐的实际执行顺序

建议后续按下面顺序逐项做：

1. 梳理聊天页和任务轮询链路
2. 梳理 Agent 返回结构与文件落盘链路
3. 梳理预览入口和静态资源组织
4. 梳理部署链路和失败处理
5. 梳理配置与权限边界
6. 梳理上下文记忆和修改计划执行
7. 再进入版本、模板、社区等能力扩展

---

## 8. 下一步建议

文档规划完成后，后续建议直接进入 P0。

P0 中最值得先做的，不是大改，而是下面这三个点：

1. 聊天任务状态流转和前端轮询逻辑收口
2. 代码生成产物目录与预览路径收口
3. 部署与下载链路的校验和异常信息收口

这三个点一旦稳住，项目整体会立刻更像一个可持续迭代的产品，而不是“功能很多但容易脆”的学生项目。

---

## 9. 备注

本路线图是基于当前仓库代码现状整理的执行规划，不是否定现有设计，而是为了：

- 保留已经做出来的成果
- 降低后续重构成本
- 让后面的完善工作更有顺序
- 避免同时开太多战线

后续如果进入具体实现阶段，可以继续拆成：

- `P0-1 主链路加固`
- `P0-2 预览与部署治理`
- `P1-1 配置治理`
- `P2-1 Agent 协议收口`

按小步提交推进。
